RES-friendly RDF

This volume of Inside Acropolis is about preparing data for indexing by RES. As well as providing a general reference, it works through a detailed example of converting some of a single web page to RDF. (Conversion of an existing web site to RDF is a common use case among organisations just starting to publish RDF.)

The example page is on the website of an imaginary castle museum, The Museum of The Castle of Otranto. Like many museum websites, it contains images, articles and videos about about the museum and its collections. Museum staff want this data to be indexed by RES so that it can be used by teachers, students and academics. As a starting point, they want to create some RDF from the page http://castleofotranto.org.uk/items/the-helmet, about a golden helmet in the museum's collection.

The rest of this volume focuses on converting this web page (and similar content) to RDF, making it RES-friendly along the way. This means using data structures which RES can index effectively, to make data more visible and accessible to RES users. The guidance given is more widely applicable to Linked Open Data (LOD), too: making your data RES-friendly helps it adhere to best practices for publishing LOD.

This is a reference guide to converting existing content into a minimal RDF dataset which can be effectively indexed by RES. It doesn't explain how to do an exhaustive conversion of all of an organisation's data, or even all of one web page: it just covers the minimum work required to offer useful data to RES. This bypasses important topics like how an organisation should decide on a license for its data, manage it, host it etc.; but these issues can be discussed with the RES team if you need help.

The museum's web page

The example museum web page has this content:

  • Identifier: t0001 (this identifier is from the museum catalogue)
  • Label: The Golden Helmet
  • Physical description: Large gold helmet
  • Dimensions: 3m diameter
  • Materials: 18 carat gold
  • Origin: Created in the 17th century by an unknown Spanish armourer.
  • History: The helmet spontaneously fell from its mount on a wall in the main hall in 1529; Conrad, son of Lord Manfred Otranto, was due to wed Isabella that day, but was instead crushed to death by the falling helmet. The helmet was then lost for many years, until, in 1749, it was rediscovered by archaeologist Fuchsia Otranto in the ruins of the old chapel.
  • Display location: On permanent display in the Arms and Armour Wing.
  • A digital photograph of the helmet, taken by Vlad Dracula in 2016.

By the end of this volume, the data on this page will have been converted to machine-readable RDF. All of the RDF examples are presented in Turtle format. For the sake of brevity, the RDF namespaces and corresponding prefixes used in the examples are summarised in an appendix and not included in the examples.

If publishing data for ingestion by RES, your RDF should be in either RDF/XML or Turtle format (or both). These are the formats currently supported by RES.

What does RES want?

RES is primarily a high-level index of the data it has ingested. It is only interested in a few key types of resource, properties and relationships. If you want to publish data which RES can index effectively, it helps to understand how the index divides the world into four broad categories of resource:

  1. Real-world things (physical and abstract; people, physical objects, places, events etc.)

  2. Creative works (novels, paintings, songs, photographs etc.)

    Note that creative works are a subclass of real-world things. Unlike real-world things, they are typically about (depict, describe, have as a subject) other real world things. By contrast, a building or a person isn't normally about anything else.

  3. Documents about things

    Examples in the real world are entries in a library catalogue, or the text on a card next to a museum exhibit. This includes documents about real-world things. (Note that for a web page about some thing, the text of that page is the document, in isolation from its representation as HTML.) RES will only index data in documents which are openly-licensed.

  4. Digital assets (digital images, text files, videos, audio files)

    These are digital representations of things. RES is mostly interested in files which represent creative works: for example, a digital image file representing a photograph; a PDF file representing a novel; or an audio file representing a sound recording.

Extracting things from the web page

Using the four categories above, the museum identifies a minimal set of resources which are useful to RES:

  1. The golden helmet

    This is a real-world thing, the actual physical helmet in the museum's collection.

  2. The photograph taken by Vlad Dracula

    The photograph is a creative work with the helmet as its subject.

  3. A JPEG file of the photograph

    This is a digital asset which represents the photograph.

  4. One or more documents which contain the data

    These are RDF documents which describe the helmet, the photograph, and the JPEG file. They also describe themselves: e.g. their licence, author, modification date. The museum has decided to publish its documents under an Open Government licence, which is acceptable to RES. See the full list in the section about licensing.

Having decided which resources to model, the next step is to choose URIs for each of them, covered in the next section.

Choosing URIs for things

In Linked Data, HTTP URIs are used as the names for things. This can seem counter-intuitive at first: a string like http://dbpedia.org/resource/Hothouse_(novel) may immediately make you think "That's the address of a web page". Linked Data requires that you instead consider URIs as names for resources, where a resource can be literally any thing. In the same way that Hothouse is the English name of this novel, http://dbpedia.org/resource/Hothouse_(novel) is an identifier for the abstract creative work which is the novel itself. Any translations of this novel are translations of http://dbpedia.org/resource/Hothouse_(novel); different editions of the novel are editions of http://dbpedia.org/resource/Hothouse_(novel); and so on. This means that, any time this URI is used, it refers to only this creative work: not a version, translation, edition, format or other variation of it, or any other thing which shares the name "Hothouse".

URIs for the museum's things

In the case of the museum example, each thing to be modelled needs a unique URI which can be hosted on the museum's web site. The museum staff decide to put their RDF resources on a /data/ sub-path of the main site, at http://castleofotranto.org.uk/data/. They use a further URI path segment to divide the site by type of data: any physical items in the museum are under /data/items/. (If you have built RESTful web services, this strategy should be familiar.)

The next part of the URI is derived from the museum database's identifier for the helmet, t0001. If your dataset includes permanent, immutable identifiers (e.g. if you use a database to power your website), they make good candidates for unique keys within your URIs. If you don't have immutable identifiers already, create them if at all possible. Ideally, they shouldn't be based on categories (as items may get re-categorised) or proper names (as languages, names, authorities etc. change over time, too).

Finally, because the resource they're creating stands in for an object in the real (rather than digital) world, they append #id onto the end of it. The final URI for the resource representing the helmet is:

http://castleofotranto.org.uk/data/items/t0001#id

Why #id? This is down to the types of resource that can be identified using URIs, and how those resources are accessed on the web. Recall that the museum is creating resources for things which exist outside the web (also known as non-information resources). The "essence" of these things can't be delivered over the web: if I put the URI http://castleofotranto.org.uk/data/items/t0001#id into my web browser, I wouldn't expect the actual helmet to be returned to me; at best, I would expect a representation of the helmet (e.g. some HTML or RDF). HTML, RDF and similar digital resources can be delivered over the web, making them information resources.

The #id is a way to deal with this: because the fragment part of a URI is ignored by the server, requesting the non-information resource http://castleofotranto.org.uk/data/items/t0001#id would actually return the information resource at http://castleofotranto.org.uk/data/items/t0001 (no #id); in this case, this resolves to an RDF representation of the helmet.

This is a Linked Data convention: when the URI of a non-information resource is requested, a server should deliver the information resource which describes it. This is known as dereferencing the URI (see Dereferencing HTTP URIs and Cool URIs for the Semantic Web).

Information and non-information resources map to RES's four categories as follows:

  • Real-world things and creative works are non-information resources.
  • Documents and digital assets are information resources.

So far, the museum has decided on the URIs (names) for two of the things in its data:

  1. http://castleofotranto.org.uk/data/items/t0001#id - the physical helmet (a real-world thing).
  2. http://castleofotranto.org.uk/data/items/t0001 - the description of the helmet (a document).

Following the principles above, and adding a further sub-path for /photos/, the museum decides on these URIs for its other resources:

  1. http://castleofotranto.org.uk/data/photos/p0001#id - the photograph of the helmet (a creative work).
  2. http://castleofotranto.org.uk/data/photos/p0001 - the description of the photograph of the helmet (a document).
  3. http://castleofotranto.org.uk/images/the-helmet-photo.jpg - the digital file representing the photograph (a digital asset); note that the existing URI for the image embedded in the web page is used.

Determining URIs for things you're publishing data about is an important step: by creating a URI for a thing, you have essentially turned it into a resource which can be published as Linked Open Data. These resources will be the subjects of RDF statements in your data, as shown in later sections.

Using existing URIs with content negotiation

The museum's resource URIs are hosted on a sub-path of the website, /data/, rather than sharing the same URIs as the existing web pages. Readers familiar with Linked Data will doubtless be familiar with content negotiation, where the same URI can return different representations, according to the preferences of the client. So why doesn't the museum reuse existing website URIs in tandem with content negotiation to serve HTML or RDF? In practice, this would work as follows: 

client requests http://castleofotranto.org.uk/items/the-helmet and wants HTML
  --> respond with HTML about the helmet

client requests http://castleofotranto.org.uk/items/the-helmet and wants RDF
   --> respond with RDF about the helmet

Details about how to implement this aren't covered here, but see Cool URIs for the Semantic Web for an overview.

There are several reasons why an organisation (like the museum) might not deliver multiple representations from a single URI, such as:

  1. Their hosting infrastructure may make it difficult to deliver different resources from a single URI. This can sometimes be due to the website implementation, particularly if based on an old content management system with opaque URLs. In such cases, it may not be feasible to map from the URL of each web page to its corresponding RDF URI.
  2. It's often simpler (and sometimes safer) to publish a data site in parallel with an existing website without touching existing infrastructure. Publishing on a sub-path (or even a separate sub-domain) can allow an organisation to safely experiment without risking existing assets. It may also allow a wider choice of implementation technology (e.g. using a cloud provider as opposed to working through an in-house web team).

Where it is feasible to use the same URIs for HTML and RDF representations (maintaining the One Web principle), organisations should strive to do it. However, doing so can be very technical, involved, and even costly. Using a sub-path or sub-domain is a quick way to get started without having to deal with these issues immediately.

Even in cases where an organisation has a separate URI space for its data, HTML and RDF can still be associated using <link> elements in the HTML. This enables a client accessing web pages to find equivalent/related RDF resources if desired.

RDF modelling for RES

This section covers how to create RDF resources for RES by following these guidelines:

  • Use notable types
  • Add basic descriptive data
  • Link and describe digital assets
  • Link documents to their formats
  • Link resources to other datasets
  • Add licensing statements

Use notable types

RES takes rdf:type statements about resources and maps them to one of a small number of generic classes. These classes are used to group similar resources together in partitions to assist discovery. The full list of partitions can be browsed at http://acropolis.org.uk/.

Inclusion of a resource in a RES partition depends on it having a specific type, or a type which can be mapped to that specific type. The list below shows each partition and the type(s) a resource should have to be put into it:

PartitionRDF types
Agents (non-person entities with agency) foaf:Agent
Audiences odrl:Group
Collections dcmitype:Collection
Concepts skos:Concept
Creative works (e.g. novels, plays, articles, paintings, photographs) frbr:Work (generally), schema:Photograph for photographs, schema:AudioObject for sounds
Digital assets (including documents) foaf:Document for documents, DCMI Types for other types of asset
Events event:Event
Groups foaf:Group
People (i.e. real people who are living or dead, and fictional characters) foaf:Person
Physical things crm:E18_Physical_Thing (generally), crm:E22_Man-Made_Object for human-made things
Places geo:SpatialThing

It's not essential for a resource to end up in a partition, but if it does, it will be more visible to a user browsing RES's index.

Adding types to the museum's resources

Returning to the example of items in the museum, the helmet is a physical thing. To be included in the physical things partition, a resource must either have the type crm:E18_Physical_Thing or a type which maps onto it. (RES's mappings between types are in Acropolis' rulebase configuration.)

One type which is mapped to crm:E18_Physical_Thing is crm:E22_Man-Made_Object, so the museum applies that to the helmet resource. This is a term from the CIDOC-CRM vocabulary, which was designed for cultural heritage data, making it ideally suited to describing things in a museum.

The museum's other resources are assigned types to slot them into RES's partitions, yielding the following RDF: 

# the helmet - will go in "Physical things"
<http://castleofotranto.org.uk/data/items/t0001#id>
  a crm:E22_Man-Made_Object .

# the document about the helmet - will go in "Digital assets"
<http://castleofotranto.org.uk/data/items/t0001>
  a foaf:Document .

# the photograph - will go in "Creative works";
# note that schema:Photograph is used as this maps to frbr:Work
<http://castleofotranto.org.uk/data/photos/p0001#id>
  a schema:Photograph .

# the document about the photograph - will go in "Digital assets"
<http://castleofotranto.org.uk/data/photos/p0001>
  a foaf:Document .

# the digital file representing the photograph - will go in "Digital assets";
# see the section below about RES's special treatment of DCMI Type resources
<http://castleofotranto.org.uk/images/the-helmet-photo.jpg>
  a dcmitype:StillImage .

The importance of DCMI Type resources

The museum's digital asset was assigned the type dcmitype:StillImage in the preceding RDF. The main purpose of this, rather than slotting the resource into one of RES's partitions, is to make it discoverable via a related media search. This type of search enables a user to find things which have related media files (a common use case for applications powered by RES). Only resources which have a type from the DCMI Type vocabulary will be returned for this type of search.

Here's an outline of the logic behind related media searches: 

If resource A is associated with resource B (via any predicate)
and resource B has the type dcmitype:T
then a search for resources with related media of type dcmitype:T will return A

As a concrete example, consider this Acropolis API query with a media query string variable:

http://acropolis.org.uk/?q=dench&media=http://purl.org/dc/dcmitype/StillImage

This returns all resources with a property value matching the string "Dench", where those resources are related to other resources which have the type http://purl.org/dc/dcmitype/StillImage.

When modelling your data, you should add DCMI Types to your digital assets to facilitate related media searches, using this table for guidance:

Type of fileExample content typeDCMI Type to use
Imageimage/jpeg, image/gif, image/pngdcmitype:StillImage
Videovideo/mp4, video/oggdcmitype:MovingImage
Audioaudio/ogg, audio/x-wav, audio/mpegdcmitype:Sound
Texttext/plain, text/html, text/turtle, application/pdfdcmitype:Text

Add basic descriptive data

The next step is to add some basic descriptive data to your resources, such as labels and descriptions. Without these, your resources will still be in the index, but a search on the Acropolis API won't find them.

At this point, it's also useful to add any basic metadata you have to the RDF, such as the creator of a creative work, the date when something was created, any museum identifiers, ISBNs or similar etc.

To add these basic descriptive statements, use the RDFS (prefix: rdfs), Dublin Core (DC) Terms (prefix: dcterms) and FOAF (prefix: foaf) vocabularies. The table below shows some of the ways you can convert existing data to RDF statements:

Type of dataExample RDF statement
Resource R has a title (e.g. a proper name) R dcterms:title "Mr. Ghost Goes To Town"@en-gb
Resource R doesn't have a proper name but has a label R rdfs:label "The desiccated hand of a thief"@en-gb
Resource R has a long description

R dcterms:description "A melting watch draped over a tree"@en-gb

R rdfs:comment "On loan from Babel"@en-gb

Resource R has a summary/synopsis R dcterms:abstract "This documentary is about waiting"@en-gb
Resource R has an identifier specific to its dataset (e.g. a unique catalogue number in a museum collection) R dcterms:identifier "x9838-aaaa"
Resource R has a creator (e.g. author, artist, developer, designer) C (where C can be a literal string or another resource) R dcterms:creator C
Resource R depicts or is about resource S R dcterms:subject S
Resource R has a related web page W (e.g. a page about resource R, or about a topic closely-related to it) R foaf:page W

rdfs:label, dcterms:title, rdfs:comment and dcterms:description are particularly useful, as Acropolis uses these for its human-readable description of a resource, as well as for free-text searches. For example, this resource for Twelfth Night gets its title "Twelfth Night" from rdfs:label and its longer description from rdfs:comment.

The DC Terms, RDFS and FOAF vocabularies are very general, widely-used, and well understood. If possible, terms from these vocabularies should be your first choice for specifying the basic properties of your resources. However, where there are no suitable terms in these vocabularies, terms from other vocabularies can be used instead. This is where data modelling becomes a craft rather than a science: choosing relevant vocabularies and applying them correctly is a skill in itself, and can be daunting.

One way to approach the task is to use a very generic but broad vocabulary, such as schema.org. This provides a wide range of different terms for describing common types of resource, and covers a considerable range of modelling tasks.

However, schema.org can be too generic for more specialised modelling. If you find that you need a more expressive vocabulary, this list of commonly-used vocabularies provides some alternatives.

One final option is to create your own vocabulary, building on top of existing ones. This option is too complex to cover in this book, but the the RES team are happy to help if you want to go down this route.

Adding basic descriptive data to the museum's resources

Considering the museum's resources again, the following RDF can be extracted from the text on the web page:

Link and describe digital assets

We've already seen how creative works are modelled using resources which stand in for the abstract creative work in the real world. This resource is distinct from any realisations of the work in physical or electronic versions. For example, the abstract novel Hothouse is a different thing from a printed copy of that novel, which is different again from an ebook copy of it.

In the context of publishing RDF, electronic files representing a creative work are information resources which can be requested over the web. These files are especially important to RES, as applications powered by RES often expose openly-licensed media to users in education. Consequently, the RES index has special mechanisms for detecting electronic media files and indexing them; these in turn rely on specific links between creative works and digital assets.

In addition, extra statements about the characteristics of the digital asset, such as its size and format, can help those wanting to use it.

Link creative works to digital assets

RES looks for three broad classes of electronic media, finding them via particular RDF predicates in the data:

Type of mediaDescriptionRDF predicateExample
Direct content Files which can be embedded directly in applications, e.g. with a <video>, <object> or <img> element in a web app. Files may be JPEG images, WAV audio, MP4 videos etc. Direct content should require no authentication. mrss:content Crackerjack image
Embeddable player An HTML page (typically) which embeds some media, and which may provide controls for playing it and/or some metadata about it. This page can be embedded in other applications (e.g. using an <iframe> in a web app). An embeddable player may prompt for authentication. mrss:player Shakespeare's Globe
Standalone player Similar to an embeddable player (displays media plus controls and metadata). Unlike an embeddable player, a standalone player should not be embedded directly into other applications. Instead, a user accessing the standalone player should see the player page open in their browser (outside the app). A standalone player may prompt for authentication. foaf:page iPlayer page

The term player applies to static media (like images) as well as dynamic media (audio and video), beyond its usual connotations.

Note that Acropolis doesn't require any authentication and relies on the server hosting the media to enforce this. However, even in cases where media require no authentication, their copyright must be respected. Again, it is up to the data publisher to assign copyright to their media to allow reuse (see the section on licensing).

Describe digital assets

If a developer wants to incorporate a digital asset into an application, knowing its content type and dimensions can help them decide how best to incorporate it. For example:

  • A document viewer is about to load an ebook. The RDF states that there are two versions available, a MOBI file and an EPUB file. As the viewer can only load EPUB files, it chooses this one.
  • An Android app is about to load an image file. The layout has a maximum width of 800px for images, and the RDF states that there are two versions of the image, one 800px wide and the other 3200px wide. The app saves some bandwidth by downloading the smaller image.

Note that format and size data is especially useful where there are multiple files relating to a creative work: it helps an application decide which file best suits its needs. See the section about creative works with multiple versions for more details.

As a minimum, it's recommended that the RDF for a digital asset describes its size (file size, image dimensions) and format. The following vocabularies can be used to do this:

  • The EXIF vocabulary can be used for image or video dimensions.
  • DC Terms can be used to specify an asset's format.
  • The Ontology for Media Resources can be used for video and audio file duration. (It also has a range of other useful terms for more detailed media file modelling.) Alternatively, schema.org has several terms for describing media objects.

The table below shows some examples of predicates you can use:

PropertyPredicate to useExample
Date created (in ISO 8601 combined date and time format) schema:dateCreated <resource> schema:dateCreated "2007-04-05T14:30+01:00"
Date modified (in ISO 8601 combined date and time format) schema:dateModified <resource> schema:dateModified "2017-05-09T15:20+01:00"
Dimensions - height (in pixels) exif:height <resource> exif:height 800
Dimensions - width (in pixels) exif:width <resource> exif:width 600
Duration (in seconds) ma:duration <resource> ma:duration 3600.0
Duration (in ISO 8601 duration format) schema:duration <resource> schema:duration "PT0M29S"
Format; see (1) below dcterms:format <resource> dcterms:format <http://purl.org/NET/mediatypes/audio/wav>

(1) The dcterms:format statement uses a value from the media types vocabulary. To determine this value, append the content type of the digital asset to http://purl.org/NET/mediatypes/ and use that as the object in the statement. Some examples:

  • AVI: http://purl.org/NET/mediatypes/video/avi

  • HTML: http://purl.org/NET/mediatypes/text/html

    Note that the content type for a web player will typically be set to this, rather than to the content type of the media in the player.

  • JPEG: http://purl.org/NET/mediatypes/image/jpeg
  • WAV: http://purl.org/NET/mediatypes/audio/wav

Linking and describing the museum's media

The museum web page has spawned a single creative work, the photograph of the helmet, with a single image file that represents it, http://castleofotranto.org.uk/images/the-helmet-photo.jpg. As this is a JPEG file which can be directly embedded in an application, the appropriate predicate to link them is mrss:content. The museum also includes some metadata about the dimensions and format of the image file, derived via a script which reads them from the file.

The resulting RDF is: 

# the photograph (creative work)
<http://castleofotranto.org.uk/data/photos/p0001#id>
  mrss:content <http://castleofotranto.org.uk/images/the-helmet-photo.jpg> .

# the file representing the photograph (digital asset)
<http://castleofotranto.org.uk/images/the-helmet-photo.jpg>
  exif:height 979 ;
  exif:width 1024 ;
  dcterms:format <http://purl.org/NET/mediatypes/image/jpeg> .

A note on multiple files

In cases where a publisher has multiple files representing a creative work, it's simple to include additional resources for them. For example, if the museum published a quarter-size digital image of the helmet photograph, it could add a resource like this: 

# the photograph; note the additional mrss:content
<http://castleofotranto.org.uk/data/photos/p0001#id>
  mrss:content <http://castleofotranto.org.uk/images/the-helmet-photo-quarter.jpg>,
    <http://castleofotranto.org.uk/images/the-helmet-photo.jpg> .

# the smaller file
<http://castleofotranto.org.uk/images/the-helmet-photo-quarter.jpg>
  exif:height 245 ;
  exif:width 256 ;
  dcterms:format <http://purl.org/NET/mediatypes/image/jpeg> .

The same approach could be used to add resources for digital assets in different formats, e.g. for a PNG version: 

# the photograph; note the additional mrss:content
<http://castleofotranto.org.uk/data/photos/p0001#id>
  mrss:content <http://castleofotranto.org.uk/images/the-helmet-photo-quarter.jpg>,
    <http://castleofotranto.org.uk/images/the-helmet-photo.jpg>,
    <http://castleofotranto.org.uk/images/the-helmet-photo.png> .

# digital file in PNG format
<http://castleofotranto.org.uk/images/the-helmet-photo.png>
  exif:height 1024 ;
  exif:width 979 ;
  dcterms:format <http://purl.org/NET/mediatypes/image/png> .

This demonstrates the benefit of adding a separate schema:Photograph resource: it can hold all of the statements which apply to all of the formats and sizes (e.g. the photographer, the date of the photograph, its location, its title), so that they don't have to be duplicated on the digital asset resources.

Link documents to their formats

The descriptions of the things in a publisher's domain are generically called documents (see What does RES want?). Like photographs, novels, songs and other creative works, a document may be available in multiple formats. For example, a description of an item may be available as a web page, a printed leaflet, or an RDF file. In all three cases, providing the data is broadly similar, these representations with different content types are treated as formats of the document.

Each format for a document should be assigned its own resource (with its own URI); the resource should have a type of dcmitype:Text and a dcterms:format value, set as for any other digital asset. Format resources are then linked to the document via the predicate dcterms:hasFormat.

In addition, if a resource is an RDF format of a document, it should be assigned a type whose value is a URI from the Unique URIs for File Formats vocabulary. Some examples for common formats:

RDF formatURI
Turtleformats:Turtle
RDF/XMLformats:RDF_XML
N3formats:N3
N-Triplesformats:N-Triples

Documents about the primary entities in a domain are themselves creative works. However, a document is mostly just a means for a user to navigate to the primary entities. This makes it counter-intuitive to treat documents (which roughly correspond to catalogue entries or similar metadata) the same way as primary entities (like the items in a museum, paintings in a gallery, or novels in a library). For this reason, a publisher shouldn't use media links (mrss:content, mrss:player) to link documents to their digital formats and should instead use dcterms:hasFormat.

Linking formats to the museum's documents

Recall that the museum has two documents relating to the web page it's converting: the document about the photograph, and the document about the helmet. As the museum intends to publish both RDF/XML and Turtle versions of the data, each of these documents has two formats. Link resources to other datasets

The RES index combines resources from different datasets into proxy resources, so that users can access all of the data relating to a topic from one place. For example, a proxy about Shakespeare's Twelfth Night may contain data aggregated from DBPedia, the BBC Genome pages, and the BBC Shakespeare archive. (For more details about how proxies are constructed, see this blog post.)

To help aggregate your data with existing proxies, link your resources to other well-known datasets where possible. This may be more or less difficult, depending on the type of data you're publishing:

  • If your data is about things which are globally recognisable or culturally significant, like ancient civilisations or popular Victorian authors, it's simple enough to find well-known datasets to link them to. Well-known datasets like DBPedia and Wikidata contain useful resources you can link to in this case.

  • If your data is highly specialised or very specific to your organisation, it may be harder to find datasets with comparable data. Extreme examples might be data about obscure topics, like UK science fiction wargaming fanzines published in the 1980s; or data about niche products, like furniture for veterinary practices. Even in these cases, you should still be able to link (albeit in a vague way) to topics from well-known datasets (e.g. wargaming, furniture); or be able to link to specialised datasets you are aware of.

    It may even be that your organisation is the authority on a particular area of knowledge. In this case, your dataset may be the authoritative one in this field: other people may need to link to you, rather than you linking to them.

How to link to other datasets

Creating a link between your resources and resources in other datasets involves publishing statements of the form: 

<your resource URI> <links to> <other resource URI>

Deciding how to populate this statement can be determined as follows:

  1. <your resource URI>: This is a source resource in your dataset which could relate to a target resource in another dataset. Good candidates are resources which belong to one or more of the partitions RES uses (places, people, events, physical things, creative works etc.).
  2. <other resource URI>: This is the target resource you are linking to. See the notes above for help with finding resources to link to.

  3. <links to>: This specifies a link between the source and the target, which falls into one of these broad groups:

    1. Similarity: Predicates such as owl:sameAs and skos:exactMatch make statements about the identity of resources: whether they are the same, nearly the same, distinct from each other etc. Statements of similarity are particularly useful for aggregating resources.
    2. Aboutness: Predicates such as dcterms:subject and foaf:topic specify that one resource is about the other in some way.
    3. Relatedness: Many predicates can be used to capture relationships which aren't to do with similarity or aboutness. These vary in vagueness from the fairly definite dcterms:creator to the more vague dcterms:isPartOf to the pretty vague rdfs:seeAlso.

The BBC is trialling a new application, Mango, which automatically creates links between chunks of text and resources in well-known datasets like DBPedia. This may help you create the types of link outlined above, enriching your data by integrating it with the wider world of Linked Data. Depending on demand, it may be free to use for organisations working with RES. Email the RES team if you are interested in finding out more.

These groups of relationship are described in more detail in the following sections.

Similarity

Various predicates can be used to express similarity between resources, but the following are by far the most commonly-used:

  • owl:sameAs: This term expresses a strong relationship between two resources. If A owl:sameAs B, A and B are identical to each other: both resources are the same entity. This is very useful and generic, but should be used carefully: the types of the two resources should be similar, at least to the extent that they are not logically inconsistent. For example, if resource A is a frbr:Work it probably cannot be owl:sameAs resource B which is a foaf:Person.
  • skos:exactMatch (source and target are the same), skos:closeMatch (source and target are nearly the same): These terms link two Simple Knowledge Organisation System (SKOS) concepts (i.e. resources with the type skos:Concept). This makes the terms less generic, but is especially useful when working with datasets based on SKOS.

Aboutness

A variety of predicates can be used to express an about relationship between resources, but the following are commonly used:

  • dcterms:subject; see these examples.
  • foaf:topic (or its inverse, foaf:page); note that FOAF predicates are defined as relating documents to things they are about, so dcterms:subject is preferable in most contexts.
  • foaf:primaryTopic for the main topic of a resource (c.f. foaf:topic).

Relatedness

Numerous other predicates can be used to express relationships between resources in your dataset and others, including the predicates we've seen earlier in this book (e.g. dcterms:creator).

rdfs:seeAlso is a particularly useful (though slightly indiscriminate) term in this context. It expresses a general "you might also be interested in" relationship between two resources. It makes no statement about whether two resources are equivalent or whether one is directly about the other. However, if resources A and B are both related by rdfs:seeAlso to resource C, it's likely that A and B are also relevant to each other in some way.

Linking the museum's resources to other datasets

Looking at their resources, the museum identifies a couple of opportunities for linking:

  • The helmet appears in the novel The Castle of Otranto, which has a DBPedia resource at http://dbpedia.org/resource/The_Castle_of_Otranto. The appropriate relationship is rdfs:seeAlso, as the DBPedia resource is pertinent to the helmet but isn't the same thing as the helmet.
  • The person who took the photograph, Vlad Dracula, is the same person as http://dbpedia.org/resource/Vlad_Dracula (the photograph was literally taken by Dracula). In this case, the museum can link directly to the DBPedia resource, replacing the previous "Vlad Dracula" value.

The museum adds these statements to their RDF: 

<http://castleofotranto.org.uk/data/items/t0001#id>
  rdfs:seeAlso <http://dbpedia.org/resource/The_Castle_of_Otranto> .

<http://castleofotranto.org.uk/data/photos/p0001#id>
  dcterms:creator <http://dbpedia.org/resource/Vlad_Dracula> .

These additions make the museum's data more discoverable, as they link it to resources in well-known datasets. For example, once these statements are indexed by RES, anyone searching for http://dbpedia.org/resource/Vlad_Dracula will be able to follow links from that URI to the museum's photograph of the helmet.

Re-using existing resources

What if the helmet already had a DBPedia resource, e.g. http://dbpedia.org/resource/The_Golden_Helmet_Of_Otranto? (It doesn't, but it could.) Why does the museum need to publish any data at all about the helmet? Couldn't it just replace all the references in its RDF to http://castleofotranto.org.uk/data/items/t0001#id with references to the DBPedia URI instead? i.e. 

<http://castleofotranto.org.uk/data/photos/p0001#id>
  a schema:Photograph ;
  dcterms:subject <http://dbpedia.org/resource/The_Golden_Helmet_Of_Otranto> .

There is nothing to stop the museum doing this: anyone can say anything about anything in Linked Data. Reusing this resource would also reduce the amount of data the museum has to publish: the golden helmet resource, http://castleofotranto.org.uk/data/items/t0001#id, could disappear from the museum's dataset.

What would be lost by doing this, though? Imagine the museum has some additional data about the helmet which isn't published anywhere else: perhaps ownership data, e.g. "Was once owned by Victor Frankenstein". This is valuable, and the museum should publish it. So where should the museum make this statement? It has several choices:

  1. Publish its own resource for the helmet (with the ownership data) and link it to the DBPedia resource: 

    <http://castleofotranto.org.uk/data/items/t0001#id>
  crm:P51_has_former_or_current_owner <http://dbpedia.org/resource/Victor_Frankenstein> ;
  owl:sameAs <http://dbpedia.org/resource/The_Golden_Helmet_Of_Otranto> .

    (Note the use of another CIDOC-CRM predicate, crm:P51_has_former_or_current_owner, to specify ownership of the helmet.)

  2. Publish the ownership data in the form of additional statements about the DBPedia resource: 

    <http://dbpedia.org/resource/The_Golden_Helmet_Of_Otranto>
  crm:P51_has_former_or_current_owner <http://dbpedia.org/resource/Victor_Frankenstein> .

    The disadvantage of this approach is that someone accessing the DBPedia resource won't get the ownership statements added by the museum. This is because those statements won't be part of the RDF document returned by http://dbpedia.org/resource/The_Golden_Helmet_Of_Otranto, as they will be published on the museum's site and not by DBPedia. The museum will also have to decide where to publish these statements; for example, it could include them with the RDF about the photograph of the golden helmet.

  3. Add the ownership data to DBPedia and don't publish any statements about the helmet at all. Because DBPedia is extracted from Wikipedia, the museum could modify Wikipedia's entry for The Golden Helmet of Otranto to (eventually) change the data in DBPedia. The drawback to this approach is that DBPedia may ignore the new ownership data or treat it incorrectly, which may mean that the data doesn't make appear correctly in DBPedia.

    Where your additional data fits snugly into Wikipedia's infobox format (which DBPedia uses for RDF extraction), it is definitely worth considering adding data directly to Wikipedia, rather than publishing it yourself. DBPedia has rigorous processes for extracting data from infoboxes in particular, and should do a decent job of creating useful RDF from any data you add there.

In summary, even if your resources exist in other datasets, you may have additional data which isn't published anywhere else. In this case, it makes sense to create your own resources (with their own URIs). Your unique data can be attached to those resources; then you can create links to corresponding resource(s) in other dataset(s), using similarity predicates. Once your resources are aggregated by RES, your unique data will then augment resources from other datasets which don't have it.

Add licensing statements

The RES crawler is responsible for harvesting data from a publisher's Linked Data site. The data it harvests is then ingested, transformed and re-published as an openly-licensed index. For this to work, the data must be licensed in a way which enables re-purposing; and statements about the licensing must be accessible to the RES crawler in the data itself.

Returning to RES's four main categories of resource, the following types of licensing statement can be applied:

  1. Documents must have licensing statements. Without these, RES won't be able to index them.
  2. Real-world things typically don't need licensing: it's impossible to copyright a person or building, for example, so there's no need to license their reuse. (Though a person's name or building design could be copyrighted.)
  3. Creative works usually have licensing; for example, the copyright on a novel or photograph defines who can reproduce it. However, RES doesn't need to know about this to crawl the data.
  4. Digital assets often have licensing; for example, copyright on a digital file allows reuse in specific contexts. (Usually, the copyright of a digital file matches the creative work or other resource which it represents.) Again, RES doesn't need to know about this to crawl the data.

The crawler only needs licensing statements on the data documents. Licensing on other resources may be included in the index after crawling, but isn't necessary for the data to be crawled.

If your digital assets can't be openly published or reused (e.g. they should only be used in educational contexts or in the UK), you will need to protect them yourself. The standard approach to doing this is to implement authentication or similar constraints in a player, then embed the digital asset inside that.

Structure of licensing statements

A licensing statement for a resource has this general form: 

<resource URI> <licensing predicate> <licence URI>

The resource URI is the URI of the resource for which you are specifying the licence; this will usually be a document, but could be a creative work or digital asset. The licensing predicate and licence URI are explained in the next two sections.

Bringing it all together
